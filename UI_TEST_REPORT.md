# Отчёт исследовательского UI-тестирования ShlapaBank (Hat Bank)

**Дата тестирования:** 28 февраля 2026  
**Метод:** Исследовательское тестирование  
**Окружение:** Desktop (1280×800), Mobile (375×812), браузер Chromium

---

## 1. Резюме

Проведено исследовательское тестирование веб-приложения ShlapaBank. Выявлены ошибки в отображении, логике и UX. Swagger и Shlapa Cheat Window оставлены по замыслу (см. README).

---

## 2. Критические дефекты (исправлены)

### 2.1. Swagger и Shlapa Cheat Window
**По замыслу:** оставлены в интерфейсе (README: «Swagger доступен прямо в интерфейсе», «Читерская шляпа» для подготовки тестовых данных).

### 2.2. ~~Пустая форма «Между своими счетами»~~ (уточнено)
- **Где:** Дашборд → Переводы → «Между своими счетами»
- **Описание:** При выборе «Между своими счетами» справа от кнопок остаётся пустое место. Форма (поля «Откуда», «Куда», «Сумма») не отображается.
- **Риск:** Невозможно выполнить перевод между своими счетами — критический сбой функционала.
- **Рекомендация:** Проверить логику отображения формы и условия её показа.

## 3. Ошибки отображения

### 3.1. Шахматный фон у логотипа-шляпы
- **Где:** Страница входа, правый верхний угол
- **Описание:** Изображение шляпы отображается на шахматном (прозрачном) фоне, типичном для редакторов изображений.
- **Риск:** Непрофессиональный вид, ощущение «сырого» продукта.
- **Рекомендация:** Экспортировать изображение с корректным фоном (например, совпадающим с фоном страницы) или в формате, поддерживающем прозрачность (PNG с альфа-каналом).

### 3.2. Несоответствие label и placeholder на странице входа
- **Где:** Поле логина на странице входа
- **Описание:** `aria-label` / `name` — «Телефон или почта», placeholder — «Логин».
- **Риск:** Путаница для пользователей и скринридеров. Неясно, что именно вводить.
- **Рекомендация:** Привести label и placeholder к одному значению (например, «Логин» или «Телефон или почта»).

### 3.3. Неверный title на странице регистрации
- **Где:** Страница `/register`
- **Описание:** В `<title>` страницы указано «ShlapaBank — вход» вместо «ShlapaBank — регистрация».
- **Риск:** Неверная вкладка в браузере, путаница при нескольких вкладках.
- **Рекомендация:** Установить корректный title для страницы регистрации.

### 3.4. Несогласованная капитализация
- **Где:** Блок «Счета» при отсутствии счетов
- **Описание:** Текст «Нет Активных Счетов. Откройте Первый Счет.» — смешение стилей (каждое слово с заглавной).
- **Рекомендация:** Использовать единый стиль: «Нет активных счетов. Откройте первый счёт.» или «Нет активных счетов. Откройте первый счет.»

## 4. Логические ошибки и несоответствия

### 4.1. Перевод на тот же счёт
- **Где:** Форма «Между своими счетами» (при наличии одного счёта)
- **Описание:** В выпадающих списках «Откуда» и «Куда» при одном счёте отображается один и тот же счёт. Пользователь может выбрать одинаковые значения.
- **Риск:** API возвращает `transfer_same_account`, но UI не блокирует такой выбор заранее.
- **Рекомендация:** Либо скрывать «свой счёт» в поле «Куда», либо блокировать кнопку «Перевести» при совпадении счетов.

### 4.2. Расхождение баланса и статистики (при определённых данных)
- **Где:** Дашборд, блоки «Общий баланс», «Счета», «Статистика»
- **Описание:** При наличии операций пополнения (например, +1000 и +2000 RUB) в «Статистика» отображаются «Доходы: 3 000,00 RUB», тогда как «Общий баланс» и баланс по счёту могут показывать 0,00 RUB.
- **Рекомендация:** Проверить синхронизацию данных: общий баланс и балансы счетов должны соответствовать сумме доходов минус расходы.

### 4.3. Сообщение об ошибке при повторном вводе
- **Где:** Страница входа
- **Описание:** После ошибки «Неверный логин или пароль» при вводе новых данных и повторной отправке сообщение об ошибке исчезает только после начала запроса (кнопка «Входим...»). Желательно очищать ошибку при изменении полей.
- **Рекомендация:** Очищать сообщение об ошибке при фокусе/вводе в поля логина или пароля.

## 5. UX и удобство использования

### 5.1. Кнопка «Закрыть» рядом со счётом
- **Где:** Карточка счёта на дашборде
- **Описание:** Кнопка «Закрыть» расположена рядом с информацией о счёте. Неочевидно, закрывает ли она карточку или сам счёт.
- **Риск:** Случайное закрытие счёта, если это именно закрытие счёта.
- **Рекомендация:** Чётко обозначить действие («Закрыть счёт») и добавить подтверждение с предупреждением о последствиях.

### 5.2. Секция «Отладка» в меню «Ещё»
- **Где:** Меню «Ещё» → «Отладка»
- **Описание:** Текст «Очищает localStorage, sessionStorage и перезагружает страницу. Используйте при сбоях отображения.» — техническая информация для разработчиков.
- **Риск:** Обычный пользователь может случайно очистить данные и потерять сессию.
- **Рекомендация:** Скрыть в production или переименовать (например, «Сбросить данные» с нейтральным описанием).

### 5.3. Пустое пространство на desktop
- **Где:** Дашборд при ширине 1280px
- **Описание:** Справа от основного контента остаётся большое пустое пространство.
- **Рекомендация:** Либо использовать эту область (виджеты, детали), либо адаптировать layout под широкие экраны.

## 6. Мобильная версия (375×812)

### 6.1. Общие замечания
- Layout адаптируется под мобильный экран.
- Нижняя навигация (Главная, Платежи, Чат, Ещё) отображается.
- Карточки (баланс, счета, статистика) располагаются вертикально.

### 6.2. Потенциальные проблемы
- **Размер кнопок:** Следует проверить, что все интерактивные элементы имеют достаточный размер для нажатия пальцем (минимум ~44×44 px).
- **Горизонтальная прокрутка:** При наличии горизонтального скролла в навигации или в блоках — проверить на разных устройствах.
- **Модальные окна:** На мобильных экранах модальные окна (переводы, OTP, открытие счёта) должны корректно занимать экран и не обрезаться.

## 9. Заключение

Приложение ShlapaBank в целом выполняет базовые сценарии (вход, регистрация, навигация), но содержит ряд дефектов, которые влияют на восприятие продукта и удобство использования. Критичным является отсутствие отображаемой формы перевода между своими счетами. Артефакты разработки (Swagger, Cheat Window, Отладка) не должны быть видны конечным пользователям.

Рекомендуется устранить критические и высокоприоритетные дефекты и провести повторное тестирование.
